# 勤怠：スタッフ日次パターン上書き（振替出勤対応）実装計画

作成日：2026-06-30
対象ブランチ：`feature/attendance-enhancement-2`（`main` から分岐）

## Context（背景・目的）

勤怠の「ベース出勤・退勤時刻」は保存されず、`AttendancePayrollTimeService::resolveBaseWindow()` が
**会社カレンダーのA/B/Cパターン（全社一律）× スタッフの勤務属性 × 平日/土日** から**都度算出**している。

問題：会社カレンダーは全社共通のため、スタッフ個別の事情（会社カレンダー上は出勤日だが本人は休み、
振替として会社カレンダー上の休日に出勤）に対応できない。**振替出勤した日は会社カレンダーにパターンが
無い（pattern=null）ため `resolveBaseWindow` が null を返し、ベース時刻が出ず**、給与区分は `no_base` で
実打刻がそのまま採用される。スタッフ個別・日次でパターンを指定する仕組みが現状ゼロ。

実現したいこと：
- スタッフ事情で振替出勤した日でも、**ベース出勤・退勤時刻を正しく表示**できるようにする。
- 勤怠管理画面（`Admin/Attendance/Index`）で、**パターン未設定日に出勤している行を背景オレンジで強調**。
- その行の「詳細」モーダルで**パターン（A/B/C）を選択**でき、保存するとベース時刻が解決して表示される。

## 設計方針（最小改修・既存算出経路への割り込み）

`attendance_records` は `unique(user_id, date)` で1スタッフ1日1レコード。よって**日次の上書きパターンは
レコード自身に持たせる**のが最小・自然（新規例外テーブル不要）。
`resolveBaseWindow` のパターン解決を「**上書きパターン優先、無ければ会社カレンダー**」に変えるだけで、
給与時刻・残業・CSVの全算出が自動的に上書きパターンで再計算される（保存せず都度算出のため波及が小さい）。

## 検証（end-to-end）
- `php artisan migrate` 後、テスト用に「会社カレンダーが休日（pattern無し）の日付」に出勤打刻のある
  `attendance_record` を用意（対象ユーザーは `work_attribute_id` 設定済み、対応する `work_attribute_pattern_times`
  に A の weekday/weekend 時刻あり）。
- `/admin/attendance`（`Admin/Attendance/Index`）で：
  - その行が**オレンジ背景＋「要パターン設定」**になり、ベース出勤/退勤が空であること。
  - 「詳細」モーダルでパターン **A** を選択→保存→再読込で**ベース出勤・退勤が算出表示**され、行のオレンジが解消されること。
  - 上書きを「未設定」に戻すと再びオレンジに戻ること。
- 既存の通常日（会社カレンダーにA/B/Cあり）の表示・給与区分が変わらないこと（デグレ確認）。
- Playwright で管理画面ログイン後に実DOM（行の背景クラス・モーダルのselect）を確認、または
  `payrollPayloadForRecord` をtinkerで直接呼び `applied_pattern`/`needs_pattern`/`base_start_at` を検証。
- `php artisan test --filter Attendance`（既存勤怠テストがあれば回帰）、`npm run build`。

## 留意点
- ベース未解決の真因が「勤務属性未設定」や「pattern_times未登録」の場合は、パターンを選んでも解決しない
  （その場合はオレンジのまま）。モーダル補足文でパターン指定の役割を明示する。`shiftDiagnostics` の理由表示と整合。
- `needs_pattern` は「打刻あり＆ベース未解決」で定義。振替で**休む側**（会社カレンダー出勤日だが本人欠勤）は
  通常そもそも打刻＝レコードが無いため一覧に出ず、対応不要（今回のスコープは振替**出勤**側の時刻表示）。
- 本番反映時は京呉服平田VPSの暫定ルール（`config:cache` 禁止・コンテナ削除禁止）に従い、`php artisan migrate` 要。
