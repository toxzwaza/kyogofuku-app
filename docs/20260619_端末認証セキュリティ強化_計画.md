# 端末認証によるアクセス制限 計画

作成日：2026-06-19 ／ システム開発部門 ／ ブランチ：`feature/security-hardening`

## 1. 目的
現状ログイン画面（店舗→ユーザー選択・パスワードなし）は誰でも入れてしまう。
**承認した端末（店舗ごとに発行）からのみログイン可能**にする。未登録端末は店舗パスワードでの初回登録を必須化する。

## 2. 全体フロー

```
[初回・未登録端末]
ログイン画面 → 店舗選択 → ユーザー選択 → ★パスワード入力欄あり
  → 店舗パスワード(例: hirata1529)入力 → 端末登録(端末ID発行)
  → 端末トークンを localStorage 保存 → ログイン成功

[2回目以降・登録済み端末]
ログイン画面 → localStorage の端末トークンをサーバ検証
  → 有効 → ★パスワード欄は非表示・代わりに「端末ID: AKB-xxxx」を表示
  → 店舗→ユーザー選択 → ログイン成功

[端末管理で解除されたら]
サーバ側でトークン無効化 → 次回ログイン時に再びパスワード入力が必要
```

## 3. データ設計（マイグレーション2点）

### 3-1. `device_registrations`（登録端末）
| カラム | 型 | 説明 |
|---|---|---|
| id | bigint | |
| shop_id | FK shops | 紐づく店舗 |
| device_code | string(unique) | **画面表示用の端末ID**（例 `AKB-7F3A21`。非機密）|
| token_hash | string(unique, index) | localStorageの端末トークンの **SHA-256ハッシュ**（生トークンはDBに保存しない）|
| label | string nullable | 端末名（任意・手入力）|
| ip_address | string | 登録時IP |
| last_ip | string nullable | 最終アクセスIP |
| user_agent | text nullable | ブラウザ/OS |
| registered_by_user_id | FK users nullable | 登録時に選択していたユーザー |
| last_used_at | timestamp nullable | 最終利用 |
| revoked_at | timestamp nullable | 解除日時（null=有効）|
| revoked_by_user_id | FK users nullable | 解除者 |
| timestamps | | |

> **セキュリティ**：localStorage に保存する生トークンは機密。DBには `token_hash` のみ保存し、照合はハッシュ比較。画面に出すのは `device_code`（非機密）。

### 3-2. 店舗パスワード … `shops` にカラム追加
| カラム | 型 | 説明 |
|---|---|---|
| device_password | string nullable | **ハッシュ化**した端末登録用パスワード（店舗ごと）|
| device_password_updated_at | timestamp nullable | 変更日時 |

> 店舗ごとに設定。未設定の店舗は端末登録不可（＝その店舗ではログイン不可）。

## 4. サーバ実装

### 4-1. エンドポイント
| メソッド/パス | 認証 | 役割 |
|---|---|---|
| `GET /device/status` | guest | localStorageのトークンを検証。`{registered, device_code, shop_name}` を返す（ログイン画面の表示制御用）|
| `POST /device/register` | guest | `{shop_id, password, label?}` を検証→端末登録→`{device_token(生), device_code}` 返却（clientがlocalStorage保存）|
| `POST /login`（改修）| guest | 既存の店舗/ユーザー認証に **端末ゲート**を追加。有効な端末トークン必須 |
| `GET /admin/device-registrations` | admin+role | 端末管理ページ（一覧）|
| `DELETE /admin/device-registrations/{id}` | admin+role | 端末解除（revoke）|
| `PUT /admin/device-registrations/shop-password/{shop}` | admin+role | 店舗パスワード設定・変更 |

### 4-2. 端末ゲート（ログイン）
`LoginRequest::authenticate()` / `AuthenticatedSessionController::store()` を改修：
1. 既存の店舗・ユーザー所属チェック
2. **端末トークン検証**：リクエストの `device_token`（localStorage由来）を SHA-256 化し `device_registrations` を照合。
   - 有効（存在・未revoke・選択店舗と一致）→ OK
   - 無効/未送信 → エラー「この端末は未登録です。店舗パスワードで登録してください」
3. ログイン成功時、`last_used_at`/`last_ip` 更新、**セッションに device_registration_id を保持**

### 4-3. 解除を即時反映するミドルウェア
`EnsureDeviceRegistered`（`auth` グループに追加）：
- セッションの `device_registration_id` が有効か毎リクエスト確認。
- revoke 済み/欠落 → ログアウトしログイン画面へ（＝解除即時反映）。
- `DEVICE_GATE_ENABLED=false` の間はスルー（段階導入用フラグ）。

### 4-4. 段階導入フラグ
`config('auth.device_gate_enabled')`（env `DEVICE_GATE_ENABLED`、既定 false）。
**店舗パスワード設定が完了するまでは false** にしておき、ロックアウトを防止。準備完了後に true。

## 5. フロント実装

### 5-1. ログイン画面 `Auth/Login.vue`
- マウント時に localStorage(`device_token`,`device_code`) を読み、`GET /device/status` で検証。
  - 登録済み → パスワード欄を非表示、代わりに「**登録端末ID: {device_code}**」を表示。
  - 未登録 → ユーザーセレクトの下に **パスワード入力欄**を表示。
- ログイン時：
  - 未登録＋パスワード入力あり → 先に `POST /device/register` → 成功でlocalStorage保存 → 続けてログインPOST（device_token添付）。
  - 登録済み → そのままログインPOST（device_token添付）。
  - 失敗時はエラー表示（店舗未選択・パスワード誤り等）。

### 5-2. 端末管理ページ `Admin/DeviceRegistration/Index.vue`（新規）
- **端末一覧**：device_code / 店舗 / ラベル / IP / ブラウザ / 登録日時 / 最終利用 / 状態（有効・解除済）/ 「解除」ボタン。
- **店舗パスワード設定**：店舗ごとに現在の設定有無＋新パスワード入力で設定/変更（保存時ハッシュ化）。
- サイドバー（`useAdminNav.js` ＋ 旧系）に「端末管理」導線を追加。

## 7. 権限・セキュリティ方針
- 端末トークンはDBにハッシュ保存・画面表示は device_code。`/device/*` はレート制限。
- 端末登録/解除/店舗パスワード変更は ActivityLog に記録。
- 店舗パスワードは bcrypt。
- localStorage の性質（消去で再登録要・ハードウェア固定ではない）を許容。IP・UAを監査用に記録。

## 8. 段階導入手順（ロックアウト防止）
1. マイグレーション＋デプロイ（`DEVICE_GATE_ENABLED=false` のまま）
2. 端末管理で **各店舗のパスワードを設定**
3. 各店舗の端末で一度登録（パスワード入力）→ localStorage 保存
4. 動作確認後 `DEVICE_GATE_ENABLED=true` に切替（以降、未登録端末はログイン不可）
