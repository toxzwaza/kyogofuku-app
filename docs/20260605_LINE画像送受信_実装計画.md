# LINE 画像送受信 MVP 実装計画

- 作成日: 2026-06-05
- 対象システム: `kyogofuku-app`（Laravel）
- 起案: 村上
- 目的: 顧客との LINE 1:1 メッセージで **画像の受信・送信** を可能にする（MVP スコープ）

---

## 1. スコープ

### やる（MVP）
- LINE → システム：ユーザーが送った画像を Webhook 受信時に同期 DL し、S3 public に保存・スレッドに表示
- システム → LINE：管理画面から画像 1枚を選んで送信、LINE 上で受信可能
- 画像 1メッセージ = 1枚（LINE 仕様）
- 表示は管理画面の既存スレッドUI に画像 inline で表示

### 今回やらない（別スコープに送り）
- 動画・音声・ファイル・スタンプ・位置情報の受信
- 画像複数枚一括送信
- 画像編集（クロップ・回転等）
- グループトーク向けの画像送信（現状は 1:1 のみ）

---

## 2. LINE Messaging API 仕様（要点）

### 受信
- Webhook イベント `event.message.type = "image"` が届く
- 本体取得：`GET https://api-data.line.me/v2/bot/message/{messageId}/content`
  - Bearer 認証（既存 `LINE_MESSAGING_CHANNEL_ACCESS_TOKEN`）
  - **送信から約10分以内に取得しないと 410 Gone**
  - レスポンスはバイナリ（JPEG が多いが PNG/GIF/WebP もあり得る）

### 送信
- push API: `{"type":"image","originalContentUrl":"https://...","previewImageUrl":"https://..."}`
- **両 URL とも HTTPS 必須**、JPEG/PNG、original ≤ 1MB / 横最大1024px、preview ≤ 240px
- preview は元画像と同じでも可（クライアント側で縮小される）

---

## 3. 仕様確定事項（村上確認済み）

| 項目 | 決定 |
|------|------|
| 画像保存先 | **S3 public**（既存 MediaFile と同じ disk `s3_public`） |
| 受信時の DL タイミング | **Webhook 同期 DL**（10分タイムアウトを確実に回避） |
| スコープ | **画像の受信・送信のみ**（複数枚・スタンプ等は次回） |
| ロールバック方針 | **機能フラグで即時 OFF**＋マイグレーション down() で完全ロールバック可能 |
| デバッグ・ログ | **専用ログチャネル `line_image`** に詳細出力。例外は必ず Log::error に payload 付きで残す |

---

## 4. 設計

### 4-0. ロールバック設計（最重要）

#### 機能フラグによる即時 OFF
- `.env` に **`LINE_IMAGE_MESSAGING_ENABLED`**（既定 `true`）を新設
- `config/line.php` の `image_messaging.enabled` で読む
- **画像処理（受信・送信）の入り口で必ずこの値をチェック**
- `false` にすれば即座に旧仕様（text のみ）の挙動に戻る
- 切り替え方法：本番 `.env` で値を変更 → `php artisan config:cache` のみで反映（再デプロイ不要、数秒）

#### マイグレーション ロールバック可能性
- `media_file_id` カラム追加は `down()` で完全に DROP できるよう FK 込みで実装
- `docker compose exec app php artisan migrate:rollback --step=1` で 1コマンド戻し

#### Git ロールバック
- 機能をマージしたコミットは独立した1コミットにする（多くても2〜3に分割）
- 本番反映時に旧コミット hash を必ず記録（手順書通り `PREV_HEAD`）
- 即時：`docker compose exec app git reset --hard <PREV_HEAD>` → restart → config:cache

#### 3段階のロールバック手順
| レベル | 手順 | 所要時間 | 状況 |
|------|------|---------|------|
| **L1 機能フラグOFF** | 本番 `.env` で `LINE_IMAGE_MESSAGING_ENABLED=false` → `config:cache` → restart | 約30秒 | コードは残し、画像処理だけ無効化。テキストは継続稼働 |
| **L2 コードロールバック** | `git reset --hard <PREV_HEAD>` → restart → config:cache | 約1分 | 新カラムは残るが旧コードに戻る（参照されないので無害） |
| **L3 マイグレーションロールバック** | L2 後に `migrate:rollback --step=1` | 約1分 | DBスキーマも完全に元に戻す |

  - `MediaFile` レコードを作成（`tags=['line_inbound']`）
  - 既存の MediaFile アップロードロジック（`MediaFileController` 内）を共通化して再利用

### 4-4. Webhook 受信ハンドリング

`app/Http/Controllers/LineMessagingWebhookController.php`
- 既存の `if ($messageType !== 'text')` 分岐を分解
- `image` のとき：
  1. `LineMessageMediaStore::storeInboundImage()` を呼んで S3 へ保存
  2. `CustomerLineMessage` に `message_type=image`、`media_file_id` 付きで insert
  3. `text` カラムは null、または「[画像]」など人間可読の placeholder（運用判断）
- `text` 以外で `image` でもないもの（sticker / file / video など）は従来通り `LineUnknownInboundMessage` に保存（MVP）

### 4-5. 管理画面 送信フロー

`app/Http/Controllers/Admin/CustomerLineMessageController.php`
- `send()` メソッド拡張：
  - リクエストパラメータに `image_file` （新規アップロード）または `media_file_id`（ライブラリから選択）を許可
  - 画像指定時は：
    1. アップロードなら S3 へ保存 → `MediaFile` 作成
    2. その MediaFile の `url`（s3_public の公開URL）を `LineMessagingService::pushImageToUser` に渡す
    3. `CustomerLineMessage` に `message_type=image`、`media_file_id` 付きで保存
- バリデーション：
  - `text` and `image_file/media_file_id` のいずれか必須
  - 画像は image MIME のみ、最大1MB（LINE 制限に合わせる）、PNG/JPEG のみ
  - 1024px を超える画像は LINE 制限のため、サーバー側で自動リサイズするのが理想（後述）

### 4-6. 管理画面 UI（スレッド画面）

`resources/js/Pages/Admin/Customer/*` の LINE メッセージスレッドコンポーネント
- 受信画像メッセージ：`<img>` で inline 表示、クリックで原寸表示
- 送信フォーム：
  - 既存テキスト入力に加えて「画像を添付」ボタン
  - クリックで `<input type="file" accept="image/png,image/jpeg">` を開く
  - プレビュー表示＋削除ボタン
  - 送信時は multipart/form-data で `image_file` を投げる

### 4-7. デバッグ・エラーログ戦略（重要）

#### 専用ログチャネル `line_image` 新設
- `config/logging.php` に追加：daily ローテーション、`storage/logs/line_image.log` に出力
- 既存 `laravel.log` と混ざらず、トラブル時に grep しやすい
- 開発時は `LOG_LEVEL=debug` 推奨、本番は `info`

#### ログ出力ポイント
| タイミング | レベル | 内容 |
|----------|------|------|
| Webhook 画像イベント受信 | info | `line_message_id`, `line_user_id`, message全体（payload） |
| LINE API content 取得開始 | debug | URL, message_id |
| content 取得成功 | info | mime, content-length, バイト数 |
| content 取得失敗 | **error** | status, response body 抜粋, message_id, lineUserId, exception |
| S3 アップロード成功 | info | path, size, mediaFile.id |
| S3 アップロード失敗 | **error** | exception, target path |
| 管理画面 画像送信 開始 | info | sender_user_id, customer_id, contact_id, mediaFile.id |
| LINE push API 呼び出し | debug | request body (originalContentUrl, previewImageUrl) |
| push API 成功 | info | (LINE は body 空、200 OK のみ) |
| push API 失敗 | **error** | status, response body, request body, lineUserId |
| バリデーションエラー | warning | 失敗内容, request payload（パスワード等は除外） |
| 機能フラグ OFF で短絡 | info | どこで OFF 判定されたか |

#### try-catch 方針
- すべての外部API呼び出し（LINE Messaging API, S3）を try-catch
- catch 内で `Log::channel('line_image')->error(..., ['exception' => $e, 'context' => [...]])`
- catch 後の挙動：
  - **受信**：例外は Webhook の 200 OK は維持しつつ DB に「画像受信失敗」レコード残す（LINE 側のリトライ抑止）
  - **送信**：例外を呼び出し元に再 throw、管理画面側でユーザーに「送信失敗」表示

#### ローカル動作確認時の便利機能
- `php artisan tail line_image` 相当：`tail -f storage/logs/line_image-*.log`
- `LineMessagingService::fetchInboundImage` で Http::fake が必要なときの簡易ラッパー
- 開発専用 artisan コマンド：`php artisan line:test-fetch-image {messageId}` で本番 LINE API から実画像 DL テスト可（実環境でのみ）

### 4-8. 画像リサイズ（自動）

LINE は original 1024px / preview 240px / 1MB を要求。
ユーザーが大きな画像をアップロードする可能性があるため、サーバー側で自動リサイズ：
- intervention/image はおそらく導入済（既存 webp 変換で使用）
- なければ GD ベースで簡易対応
- 保存は `original`（最大 1024px / 1MB） と `preview`（最大 240px） の2バリアント
- MediaFile レコードは `original` のみ作成、`preview` は同一 path から webp で生成 or 縮小版を別 path で

**MVP 簡易判断**：MediaFile 既存ロジックの `webp_path` を流用し、`previewImageUrl` には webp 縮小版 URL を、`originalContentUrl` には MediaFile.url を渡す。ただし LINE は webp 非対応のため、PNG/JPEG にリサイズして両方とも提供する。

→ 実装簡略化のため：**`originalContentUrl` と `previewImageUrl` は同一の S3 URL（リサイズ済み 1024px max）** で行く（LINE クライアントが自動縮小）。**1MB 超は自動リサイズ**。

## 7. リスク・注意点

| リスク | 対策 |
|------|------|
| LINE 10分タイムアウト | Webhook 同期 DL を採用、リトライなし（受信漏れ時は次回別画像で代替） |
| 大画像での LINE API 拒否 | 受信時：S3 にそのまま保存（LINE 側送信時に既に圧縮済み）／送信時：1MB / 1024px 超は自動リサイズ |
| S3 public バケットが意図せず公開 | 既存 MediaFile と同じ `s3_public` disk を流用、ACL は既存運用に合わせる |
| Webhook 処理時間増加 | LINE は3秒以内のレスポンスを推奨。同期DLは数百ms〜2秒程度。応答が遅延しすぎる場合は Queue 化を検討 |
| 個人情報を含む画像 | 顧客が送る画像は基本的に保管対象（顧客カルテと同等の機密管理） |
