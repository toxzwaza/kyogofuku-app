# Blade LP 予約確認メール 実装計画（2026-05-29）

別エージェントが本ドキュメントだけを読んで実装に着手できるよう、現状・要件・変更内容・テスト方針を完結にまとめる。

---

## 1. 背景・要件

### 1.1 現状の問題

イベント予約LPには **2系統** あり、確認メール送付の挙動が食い違っている：

| 系統 | ご予約者への確認メール | 店舗LINEグループ通知 | Google カレンダー連携 |
|------|----------------------|----------------------|----------------------|
| **Vue 版** (`EventReservationController::store`) | ✅ 送信 | ✅ | ✅ |
| **Blade LP 版** (`BladeReservationController::store`) | ❌ **未実装** | ✅ | ✅ |

Blade LP 経由で予約したお客様は、**予約完了の自動返信メールを一切受け取れていない**。Vue 版利用者との不公平が発生しており、機能漏れと判断した。

### 1.2 ゴール

Blade LP 経由の予約完了時にも、Vue 版と同等の確認メールを送付する。具体的には：

1. `ReservationConfirmationMail` Mailable を Blade LP の予約完了処理から呼び出す
2. 送信履歴を `Email` テーブル（`EmailThread` 紐付け）に記録する
3. 初回友だち追加導線として、未連携の予約に対して `CustomerLineLinkToken` を発行し LIFF URL をメール本文に同梱する（Vue 版と同等の振る舞い）
4. メール送信失敗時も予約処理は完了させる（Vue 版・LINE 通知と同じ try/catch ポリシー）

### 1.3 非機能要件

- メール本文・件名・スレッド管理ロジックは **Vue 版と完全に同一** にする（Mailable / Blade テンプレートを使い回す）
- Vue 版の挙動は変更しない（リファクタの副作用ゼロ）

### 2.2 sendReservationConfirmationEmail の処理フロー

```
1. $reservation->load('event', 'document');
2. LINE LIFF URL を組み立て:
   - EventLineShopResolver で shop_id を解決
   - shop_id があれば CustomerLineLinkToken を発行（30日有効、suggested_label = '本人'）
   - $lineLiffUrl = config('app.url') . '/line/liff/link/' . rawurlencode($token)
3. メール用ビュー変数 $mailViewData を組み立て
4. EmailThread::firstOrCreate(['event_reservation_id' => $reservation->id], ['subject' => "【{$event->title}】ご予約ありがとうございます"])
5. $mailable = new ReservationConfirmationMail($reservation, $emailThread->id, $lineLiffUrl)
6. Mail::to($reservation->email)->send($mailable)
7. raw email 文字列を buildRawEmail() で組み立て
8. Message-ID を生成（<reservation-confirmation-{id}-{ts}@{host}>）
9. Email::create([...]) で送信履歴を保存
10. Log::info で記録
```

## 3. 実装方針

### 3.1 ハイレベル設計

**Vue 版の `sendReservationConfirmationEmail` / `buildRawEmail` を Service クラスに切り出して共通化する。** Blade LP 側はそのサービスを呼ぶだけになる。

理由：
- 同じロジックを2箇所にコピーするとメンテ時の漏れリスクが高い（LINE LIFF URL の組み立て・EmailThread・Email・raw メール文字列など複雑）
- プロジェクトには既に `app/Services/Line/`, `app/Services/BladeLp/` のような層分けがあり、Service 抽出が自然
- Mailable・ビュー・Model はそのまま再利用するため変更は局所的

### 3.2 サービス名と配置

新規ファイル：`app/Services/ReservationConfirmationMailer.php`

名前空間 `App\Services` 直下（Vue 版・Blade 版どちらからも呼ばれる共通サービスのため、`BladeLp/` 配下は不適切）。

### 3.3 移行戦略

1. **Phase 1**：Service クラスを新設し、ロジックを完全コピーで移植（Vue 版は変更しない）
2. **Phase 2**：Vue 版の `EventReservationController::sendReservationConfirmationEmail` を Service 呼び出しに置き換える（同じ Service を経由するので挙動は不変）
3. **Phase 3**：Blade 版 `BladeReservationController::store()` から Service を呼び出す追加実装

各フェーズで段階的にテストを通せる構造。

## 4. 変更詳細（ファイル単位）

### 4.1 新規：`app/Services/ReservationConfirmationMailer.php`

```php
<?php

namespace App\Services;

use App\Mail\ReservationConfirmationMail;
use App\Models\CustomerLineLinkToken;
use App\Models\Email;
use App\Models\EmailThread;
use App\Models\EventReservation;
use App\Services\Line\EventLineShopResolver;
use Illuminate\Support\Facades\Log;
use Illuminate\Support\Facades\Mail;

class ReservationConfirmationMailer
{
    public function __construct(
        private EventLineShopResolver $eventLineShopResolver,
    ) {}

    /**
     * 予約者へ受付完了メールを送信し、送信履歴を DB に保存する。
     *
     * 送信失敗時は Log::error のみで例外を握りつぶさず再 throw する設計にし、
     * 呼び出し側で try/catch する（Vue 版・Blade 版で挙動を揃えるため）。
     */
    public function send(EventReservation $reservation): void
    {
        $reservation->load('event', 'document');

        $lineLiffUrl = $this->issueLineLiffUrl($reservation);

        $event = $reservation->event;
        $eventTitle = $event?->title ?? 'イベント';

        $mailViewData = [
            'reservation' => $reservation,
            'lineLiffUrl' => $lineLiffUrl,
            'lineAddFriendUrl' => (string) config('line.line_official_add_friend_url', ''),
            'reservationEmailIncludeAddFriendUrl' => (bool) config('line.reservation_email_include_add_friend_url', true),
        ];

        $emailThread = EmailThread::firstOrCreate(
            ['event_reservation_id' => $reservation->id],
            ['subject' => "【{$eventTitle}】ご予約ありがとうございます"]
        );

        $mailable = new ReservationConfirmationMail($reservation, $emailThread->id, $lineLiffUrl);
        Mail::to($reservation->email)->send($mailable);

        $rawEmail = $this->buildRawEmail($mailable, $reservation->email, $mailViewData);
        $messageId = $this->buildMessageId($reservation->id);

        Email::create([
            'message_id' => $messageId,
            'from' => config('mail.from.address'),
            'to' => $reservation->email,
            'subject' => $mailable->envelope()->subject,
            'text_body' => view('mail.reservation_confirmation_plain', $mailViewData)->render(),
            'html_body' => view('mail.reservation_confirmation', $mailViewData)->render(),
            'raw_email' => $rawEmail,
            'event_reservation_id' => $reservation->id,
            'email_thread_id' => $emailThread->id,
        ]);

        Log::info('受付完了メールを送信しました', [
            'reservation_id' => $reservation->id,
            'email' => $reservation->email,
            'email_thread_id' => $emailThread->id,
        ]);
    }

    /**
     * イベントに店舗が紐付いていれば、初回友だち追加用の LIFF URL を発行する。
     * shop が解決できない場合や CustomerLineContact 既存時は null を返す。
     */
    private function issueLineLiffUrl(EventReservation $reservation): ?string
    {
        $event = $reservation->event;
        if (! $event) {
            return null;
        }

        $shopId = $this->eventLineShopResolver->resolveShopIdForEvent($event);
        if ($shopId === null) {
            return null;
        }

        $token = CustomerLineLinkToken::generateToken();
        CustomerLineLinkToken::query()->create([
            'token' => $token,
            'customer_id' => null,
            'event_reservation_id' => $reservation->id,
            'shop_id' => $shopId,
            'suggested_label' => '本人',
            'expires_at' => now()->addDays(30),
            'created_by_user_id' => null,
        ]);

        return rtrim((string) config('app.url'), '/').'/line/liff/link/'.rawurlencode($token);
    }

    private function buildRawEmail(ReservationConfirmationMail $mailable, string $to, array $mailViewData): string
    {
        $envelope = $mailable->envelope();
        $subject = $envelope->subject;
        $from = config('mail.from.address');
        $fromName = config('mail.from.name');

        $textBody = view('mail.reservation_confirmation_plain', $mailViewData)->render();

        $rawEmail  = "From: {$fromName} <{$from}>\r\n";
        $rawEmail .= "To: {$to}\r\n";
        $rawEmail .= "Subject: {$subject}\r\n";
        $rawEmail .= "Reply-To: reply@reply.kyogofuku-hirata.jp\r\n";
        $rawEmail .= 'Date: '.now()->format('r')."\r\n";
        $rawEmail .= "MIME-Version: 1.0\r\n";
        $rawEmail .= "Content-Type: text/plain; charset=UTF-8\r\n";
        $rawEmail .= "\r\n";
        $rawEmail .= $textBody;

        return $rawEmail;
    }

    private function buildMessageId(int $reservationId): string
    {
        $host = parse_url((string) config('app.url'), PHP_URL_HOST) ?: 'localhost';
        return '<reservation-confirmation-'.$reservationId.'-'.now()->timestamp.'@'.$host.'>';
    }
}
```

#### 4.1.1 Vue 版実装との差分メモ
- 元の `buildRawEmail` は `$mailable->reservation` から view 用変数を再構築していた。Service 化で同じ `$mailViewData` を引き回せるため重複なし。
- 元の `Reply-To` ヘッダはハードコード `'reply@reply.kyogofuku-hirata.jp'`。Vue 版と完全一致のため移植時もハードコードする（環境変数化は別案件）。
- 元コードは `parse_url(..., PHP_URL_HOST)` の戻り値が null になるケースを考慮していない。Service 化に合わせ `?: 'localhost'` のフォールバックを追加。

#### 4.3.3 設計メモ
- **管理画面からの登録（from_admin）判定は不要**：`BladeReservationController` は公開LPからのみ呼ばれるため、Vue 版の `$fromAdmin` 分岐は持ち込まない。

- `app(...)` でコンテナ解決。コンストラクタ DI を追加する手もあるが、`BladeReservationController` の既存コンストラクタは `FormSchemaValidator` のみで、ここに増やすと変更点が広がるため `app()` 呼び出しに留める。

## 8. 留意事項・実装時の確認ポイント

### 8.1 EventReservation のリレーション存在確認

`$reservation->load('event', 'document');` の `document` リレーションが Blade LP 用イベントでも安全に呼べることを確認：

```bash
grep -nE "public function document\(\)" app/Models/EventReservation.php
```

リレーション未定義なら `load()` で例外になる。Vue 版で動いている実績があるため定義されているはずだが、念のため確認する。

### 8.2 Mail ファサードのキュー設定

`ReservationConfirmationMail` は `use Queueable` を含むが、現状 `Mail::to()->send()` は同期送信。**キュードライバ依存の挙動変更はしない**。

### 8.3 EmailThread のスキーマ

`EmailThread::firstOrCreate(['event_reservation_id' => ...], ['subject' => ...])` を使うため、`email_threads.event_reservation_id` に unique 制約 or 適切なインデックスが必要。既存運用で動いているので問題ないはずだが、Blade LP 経由で初めて使われる場合に備えて確認：

```bash
grep -rn "email_threads" database/migrations/
```

### 8.4 LIFF URL 生成のホスト

`config('app.url')` がローカル開発で `http://localhost` のまま運用されていると、メールに不適切な URL が入る。**Blade LP の本番テスト時は `APP_URL` が本番ドメインになっていることを確認**。

### 8.5 Mailer の `app()` 解決と DI の整合

- `BladeReservationController` 側：`app(ReservationConfirmationMailer::class)`
- `EventReservationController` 側：コンストラクタ DI

形式が違うのは意図的（既存コンストラクタの拡張を最小化）。両方とも Service Container 経由で同じインスタンスが返るため挙動は同一。テストでバインドを差し替える場合は `$this->app->instance(ReservationConfirmationMailer::class, $mock);` を使う。

### 8.6 ロギングのプレフィックス

Vue 版ログは `'受付完了メールを送信しました'`（プレフィックス無し）、Blade LP のエラーログは `'[BladeReservation] 受付完了メール送信失敗'`（プレフィックスあり）。Blade LP 側の既存ログ（Google カレンダー・LINE）が `[BladeReservation]` プレフィックス付きなので、それに合わせる。
