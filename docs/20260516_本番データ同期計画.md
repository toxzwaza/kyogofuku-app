# 本番データ → ローカル同期計画

- 作成日: 2026-05-16
- 対象システム: `京呉服平田/HP_WP/kyogofuku-app`（Laravel + MariaDB）
- 起案: 村上
- 目的: 追加開発・テスト用に本番DB＋画像ストレージを最新化する

---

## 1. 前提・方針

- **同期範囲**: DB 全64テーブル + `storage/app/public/` 画像
- **ローカルDB**: 既存 `localdb` は残し、新たに `localdb_prod` を作って併存運用
  - 切替は `.env` の `DB_DATABASE` を書き換える方式
- **本番DB接続**: SSH `160.251.139.12` → `docker exec shared_db mysqldump`（読み取りのみ）
- **画像同期**: 本番 `/srv/laravel-apps/event-reserve/app/storage/app/public/` → ローカル `storage/app/public/` を rsync
- **補正**: 取り込み後、`docs/本番同期後_店舗マスタ補正メモ.md` のSQLでGoogleカレンダーID／LINE Group IDをローカル用に置換

## 3. Phase 0: 事前準備

```bash
# 作業ディレクトリ
APP_DIR="/Users/to-murakami/業務データ/CC_会社/システム開発部門/京呉服平田/HP_WP/kyogofuku-app"
DUMP_DIR="/Users/to-murakami/業務データ/CC_会社/システム開発部門/京呉服平田/HP_WP/dump"
TS=$(date +%Y%m%d_%H%M)

# 0-1. ローカル localdb の現状バックアップ（万一に備えて）
mysqldump -h 127.0.0.1 -u to_murakami -pMurakami0819 \
  --single-transaction --default-character-set=utf8mb4 localdb \
  > "$DUMP_DIR/local_localdb_${TS}_pre_sync.sql"

# 0-2. ローカル storage/app/public のスナップショット（差分ロールバック用）
tar czf "$DUMP_DIR/local_storage_${TS}_pre_sync.tgz" \
  -C "$APP_DIR" storage/app/public
```

## 4. Phase 1: 本番DBダンプ取得

```bash
PROD_HOST="to-murakami@160.251.139.12"
TS=$(date +%Y%m%d_%H%M)

# 1-1. 本番側で .env から DB情報を確認（DB名・ユーザー・パスワード）
ssh "$PROD_HOST" "grep -E '^DB_(DATABASE|USERNAME|PASSWORD)=' /srv/laravel-apps/event-reserve/app/.env"

# 1-2. shared_db コンテナ内で mysqldump → ホスト /tmp に出力
#    ※ DB名等は前段の確認結果で置換すること
ssh "$PROD_HOST" "docker exec shared_db sh -c '
  mysqldump -u<DB_USER> -p<DB_PASS> \
    --single-transaction --default-character-set=utf8mb4 \
    --routines --triggers --no-tablespaces \
    <DB_NAME>
' | gzip > /tmp/kyogofuku_prod_${TS}.sql.gz"

# 1-3. 念のためサイズと先頭確認
ssh "$PROD_HOST" "ls -lh /tmp/kyogofuku_prod_${TS}.sql.gz && \
  zcat /tmp/kyogofuku_prod_${TS}.sql.gz | head -5"
```

> **メモ**: `--no-tablespaces` は MariaDB10.11 でも安全。`--single-transaction` でロックを避ける。

## 5. Phase 2: ダンプ転送

```bash
TS=<Phase1と同じTS>
PROD_HOST="to-murakami@160.251.139.12"
DUMP_DIR="/Users/to-murakami/業務データ/CC_会社/システム開発部門/京呉服平田/HP_WP/dump"

# 2-1. ローカルへ転送
scp "$PROD_HOST:/tmp/kyogofuku_prod_${TS}.sql.gz" "$DUMP_DIR/"

# 2-2. 本番側から削除
ssh "$PROD_HOST" "rm /tmp/kyogofuku_prod_${TS}.sql.gz"

# 2-3. ローカルで解凍
cd "$DUMP_DIR"
gunzip -k "kyogofuku_prod_${TS}.sql.gz"   # -k で .gz を残す
ls -lh "kyogofuku_prod_${TS}.sql"*
```

## 6. Phase 3: ローカルへインポート（別DB名で併存）

```bash
TS=<Phase1と同じTS>
DUMP_DIR="/Users/to-murakami/業務データ/CC_会社/システム開発部門/京呉服平田/HP_WP/dump"

# 3-1. localdb_prod を作成（既存があれば一旦DROP）
mysql -h 127.0.0.1 -u to_murakami -pMurakami0819 -e "
  DROP DATABASE IF EXISTS localdb_prod;
  CREATE DATABASE localdb_prod
    CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
"

# 3-2. インポート（serialized data 対策で --binary-mode=1）
mysql -h 127.0.0.1 -u to_murakami -pMurakami0819 \
  --binary-mode=1 --default-character-set=utf8mb4 \
  localdb_prod < "$DUMP_DIR/kyogofuku_prod_${TS}.sql"

# 3-3. インポート結果の確認
mysql -h 127.0.0.1 -u to_murakami -pMurakami0819 -e "
  SELECT table_name, table_rows
  FROM information_schema.tables
  WHERE table_schema = 'localdb_prod'
  ORDER BY table_rows DESC LIMIT 20;
"
```

## 8. Phase 5: 画像ストレージ同期

```bash
PROD_HOST="to-murakami@160.251.139.12"
APP_DIR="/Users/to-murakami/業務データ/CC_会社/システム開発部門/京呉服平田/HP_WP/kyogofuku-app"

# 5-1. 本番コンテナ内パスからホスト側パスに出してから rsync するのが安全
#    （event_app コンテナの storage/app/public をホスト側にコピー）
ssh "$PROD_HOST" "
  docker cp event_app:/var/www/html/storage/app/public /tmp/kyogofuku_storage_$(date +%Y%m%d)
  ls -la /tmp/kyogofuku_storage_$(date +%Y%m%d) | head
"

# 5-2. rsync で取得（差分のみ・公開ファイルを上書き）
rsync -avz --progress \
  "$PROD_HOST:/tmp/kyogofuku_storage_$(date +%Y%m%d)/" \
  "$APP_DIR/storage/app/public/"

# 5-3. 本番側の一時ディレクトリ削除
ssh "$PROD_HOST" "rm -rf /tmp/kyogofuku_storage_$(date +%Y%m%d)"

# 5-4. シンボリックリンク (public/storage) 再作成（必要なら）
cd "$APP_DIR"
[ -L public/storage ] || php artisan storage:link
```

> **代替**: `docker cp` を介さず、本番のホスト側マウントが分かるなら直接 rsync するほうが速い。
> `docker inspect event_app --format '{{ range .Mounts }}{{ .Source }} -> {{ .Destination }}\n{{ end }}'` で確認。

## 10. ロールバック手順

```bash
APP_DIR="/Users/to-murakami/業務データ/CC_会社/システム開発部門/京呉服平田/HP_WP/kyogofuku-app"
DUMP_DIR="/Users/to-murakami/業務データ/CC_会社/システム開発部門/京呉服平田/HP_WP/dump"

# A. .env を元に戻す（localdb に戻す）
cd "$APP_DIR"
sed -i '' 's/^DB_DATABASE=.*/DB_DATABASE=localdb/' .env
php artisan config:clear

# B. localdb_prod を捨てる
mysql -h 127.0.0.1 -u to_murakami -pMurakami0819 -e "DROP DATABASE IF EXISTS localdb_prod;"

# C. 画像を pre_sync スナップショットに戻す（必要時のみ）
rm -rf storage/app/public
mkdir -p storage/app/public
tar xzf "$DUMP_DIR/local_storage_<TS>_pre_sync.tgz" -C "$APP_DIR"
```

## 11. リスクと注意点

| リスク | 対策 |
|---|---|
| 本番DB ロック発生 | `--single-transaction` で防止。実行は混雑時間帯を避ける |
| Phase 7 で .env 切替忘れ → ローカル localdb のまま見ている | Phase 6-2 で必ず `getDatabaseName()` を確認 |
| **Phase 4 の補正忘れ → 本番Googleカレンダーへ書き込み** | Phase 6-2 で shops を必ず確認。岡山店・城東店のIDがローカル用になっていなければ作業中止 |
| ダンプサイズが大きい（activity_logs 7万 / event_utm_trackings 33万件） | gzip圧縮で転送。所要時間目安は数分〜十数分 |
| 画像が壊れる/欠落 | `Phase 0-2` のtarで戻せる |
| `config:cache` を実行 → LINE通知が壊れる | キャッシュ操作は **`config:clear` のみ** とする（既知の罠） |
