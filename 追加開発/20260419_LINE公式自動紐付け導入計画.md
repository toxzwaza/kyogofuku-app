## 1. 目的・背景・現状課題

### 1-1. 目的
LINE公式アカウント追加時に、自社の顧客情報・予約情報と **「ボタン1タップで自動紐付け」** が完了する仕組みを構築し、CRM施策（予約リマインド・キャンペーン配信・1to1サポート）を機能させる。

### 1-2. 背景
現状の導線は2段階構成。

| 段階 | 内容 | 現状の達成状況 |
|------|------|---------------|
| ① 公式アカウント追加 | 予約サイトのLIFFから友だち追加URLへ誘導 | ほぼ100%が達成 |
| ② 顧客情報・予約情報との紐付け | 別途LIFFを開いて顧客IDを入力／LINEログインで認証 | **ほぼ0%（ユーザーが意図的に行動する必要があり離脱）** |

②が達成されないため、LINE userIdと自社顧客レコードが結合されず、以下の本来の効果が得られていない。

- 予約日前日の自動リマインド送信
- セグメント配信（性別・来店履歴・嗜好）
- 個別カスタマーサポート（チャット）
- リピート促進クーポン配信

### 1-3. KPI

| 指標 | 現状 | 導入後目標（3ヶ月以内） |
|------|------|--------------------------|
| 友だち追加→顧客情報紐付け率 | 5%未満（推定） | **80%以上** |
| 紐付け完了までの平均タップ数 | 5回以上（離脱多） | **1〜2タップ** |
| LINE経由の予約リマインド到達率 | 5%未満 | **80%以上** |

## 2. 実装方式の概要図

### 2-1. 全体フロー（推奨：Account Link方式）

```
[予約完了画面]
   ↓ サーバーが linkToken（ワンタイム）を発行
   ↓ 「LINEで予約と連携する」ボタン表示
   ↓ ユーザータップ
[LINE Platform：Account Link URL]
   ↓ LINEログイン
   ↓ 公式アカウント未追加なら自動で「友だち追加」案内
   ↓ ユーザーが追加
[LINE → 自社サーバー webhook]
   ・followイベント：LINE userId 取得
   ・accountLinkイベント：linkToken と userId が同時に届く
   ↓
[サーバー]
   ・linkToken から該当する予約・顧客IDを特定
   ・customers.line_user_id = userId に更新
   ↓
[あいさつメッセージ（FlexMessage）]
   ・「○○様、連携が完了しました」
   ・LIFFボタン「予約状況を確認する」
   ↓ ユーザータップで LIFF 起動可能
[紐付け完了：以降はLINE経由でリマインド・配信が可能]
```

### 2-2. シンプル代替フロー（短期暫定案）

Account Link実装が間に合わない場合の暫定案。

```
[予約完了画面] → 友だち追加URLへ
   ↓
[公式アカウント追加]
   ↓
[あいさつメッセージ：固定]
   ・「予約と連携してください」テキスト
   ・LIFFボタン（共通URL）
   ↓ タップ
[LIFF画面]
   ・予約番号入力フォーム or 電話番号認証
   ・liff.getProfile() で userId 取得
   ・サーバーへPOST → DB照合・紐付け
```

## 3. 必要な技術要素

| 要素 | 用途 | 備考 |
|------|------|------|
| LINE Messaging API | webhook受信、メッセージ送信 | チャネル種別：Messaging API |
| LINE Login + LIFF v2 | ユーザー認証・ブラウザ内アプリ | LIFF SDK `liff.init / getProfile / closeWindow` |
| Account Link API | 自社IDとLINE userIdの紐付け | `https://api.line.me/v2/bot/user/{userId}/linkToken` |
| webhookエンドポイント | follow / accountLink イベント処理 | HTTPS必須・署名検証必須 |
| あいさつメッセージ | 友だち追加直後の自動メッセージ | LINE Official Account Managerで設定。FlexMessage対応 |
| DBスキーマ追加 | `customers.line_user_id` 列追加 | UNIQUE制約・index |
| 紐付けAPI | サーバー側エンドポイント | POST `/api/line/link` |

## 6. 想定コスト・リスク・代替案

### 6-1. コスト

| 項目 | 金額 | 備考 |
|------|------|------|
| 開発工数（内製） | 0円 | 約19人日 |
| LINE Messaging API利用料 | 0円〜 | フリープラン（月1,000通）超過時のみ従量。CRM配信件数に応じて月数千〜数万円規模を見込む |
| LIFF利用料 | 0円 | 無料 |
| サーバー追加コスト | 0円 | 既存VPS内で完結 |
| **初期費用合計** | **0円（内製の場合）** | 外注時は80〜120万円目安 |

### 6-2. リスクと対策

| リスク | 影響度 | 対策 |
|--------|--------|------|
| LINE Platform仕様変更（Account Link API廃止等） | 中 | 暫定案（LIFF + 予約番号入力）を併用設計しておく |
| webhookエンドポイントのダウン | 高 | 死活監視・自動リトライ・エラーログ通知 |
| ユーザーのLINEバージョン古い／プライバシー設定でuserId取得不可 | 低 | エラー時はカスタマーサポート連絡導線を表示 |
| 既存②導線との並走による混乱 | 中 | 旧フローは段階的に停止（リリース後2週間でクローズ） |
| 個人情報保護（LINE userIdは個人情報扱い） | 高 | プライバシーポリシー改訂・同意取得画面をLIFFに組込 |

### 6-3. 代替案比較

| 案 | 概要 | 紐付け率見込み | 工数 | 採否 |
|----|------|---------------|------|------|
| **本案：あいさつメッセージ + Account Link** | 友だち追加直後に自動紐付け | **80%以上** | 19人日 | **採用** |
| 案2：LINEログインを予約フォームに統合 | 予約時点でuserId取得 | 95%以上 | 40人日超 | 中長期検討 |
| 案3：リッチメニュー常設のみ | 自発タップ待ち | 10%未満 | 5人日 | 不採用 |
| 案4：手動紐付けキャンペーン | スタッフ対応 | 30%程度 | 継続的負荷 | 不採用 |

| 指標 | 計測方法 | 集計頻度 |

## 8. 結論

予約完了→公式アカウント追加→自動紐付けまでを **Account Link APIで一気通貫** にすることで、現状ほぼ機能していないCRM導線を一気に立ち上げる。約3週間・内製で完結し、追加コストゼロ。
