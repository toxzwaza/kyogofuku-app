# LINE 公式自動紐付け実装メモ（2026-04-20）

## 1. 概要

公式アカウント友だち追加経由のユーザー（link token を持たない素のユーザー）が、
**自分で「予約番号」または「電話番号」＋お名前カナを入力するだけで紐付けが完了する**
セルフ紐付けLIFFを新設。あいさつメッセージのFlexMessageからワンタップで誘導する。

## 4. 動作フロー

```
[ユーザー：公式アカウント追加]
   ↓
[LINE Platform → /webhook/line/messaging]
   ・follow イベント受信 → line_unknown_inbound_messages に追跡記録
   ↓
[LINE OAM の あいさつメッセージ（FlexMessage）が自動送信]
   ・「ご予約・情報と連携する」ボタン（LIFF URL）
   ↓ タップ
[GET /line/liff/welcome]
   ・liff.init() → LINEログイン → liff.getIDToken()
   ↓
[ユーザー：予約番号/電話番号 + 名前カナ を入力]
   ↓
[POST /line/liff/welcome/match]
   ・id_token 検証
   ・予約番号 → event_reservations.id で照合
   ・電話番号 → event_reservations.phone or customers.phone_number
   ・kana が指定されていれば furigana / kana と AND
   ・一致したら customer_line_contacts に upsert（line_user_id ↔ customer or reservation）
   ・初回紐付け時のみ welcome push（既存 link_welcome_text 利用）
   ↓
[紐付け完了：以降、リマインド・配信が可能]
```

## 5. 必要な環境変数（.env 追記）

```env
# あいさつメッセージ専用 LIFF ID（未設定時は LINE_LIFF_ID にフォールバック）
LINE_WELCOME_LIFF_ID=
```

既存の以下は引き続き必要：
- `LINE_MESSAGING_CHANNEL_SECRET`
- `LINE_MESSAGING_CHANNEL_ACCESS_TOKEN`
- `LINE_LIFF_ID`
- `LINE_LOGIN_CHANNEL_ID`
- `LINE_LINK_WELCOME_TEXT`（オプション、push本文）

## 6. LINE Developers コンソール側の作業

1. **LIFF アプリを新規追加**（既存「LINEログイン」チャネル配下）
   - Endpoint URL：`https://<本番ドメイン>/line/liff/welcome`
   - Scope：`profile`、`openid`
   - Bot link feature：On Air（あいさつメッセージから到達するため任意）
   - 取得した LIFF ID を `.env` の `LINE_WELCOME_LIFF_ID` に設定
2. **LIFF Endpoint URL を `/line/liff/welcome` で完全一致**にすること（ngrok 等で開発時もこれと同じパスで開く）
3. `php artisan config:clear` を本番で実行

## 7. LINE Official Account Manager 側の作業

1. ホーム → メッセージ配信 → **あいさつメッセージ** を編集
2. 既存テキストを残す or 削除
3. メッセージタイプを **Flex Message** に切り替え
4. `追加開発/あいさつメッセージ_FlexMessage.json` の内容を貼り付け
5. URI内の `{LINE_WELCOME_LIFF_ID}` を実際のLIFF IDに置換（例：`1234567890-AbCdEfGh`）
6. 保存

## 9. エラー時の挙動

| 入力 | 結果 |
|------|------|
| 該当予約・顧客なし | 404「該当するご予約・お客様情報が見つかりませんでした。」 |
| 既に同じLINEで連携済 | 200「既に連携済みです。」（already_linked: true） |
| 別の予約に同じLINEが既に紐付済 | 422「別の LINE が連携済み」 |
| LINE設定不備 | 503「設定エラー」 |
| id_token 検証失敗 | 401「LINE の認証に失敗しました」 |

## 10. セキュリティ・プライバシー考慮

- LINE userId は LINE 内識別子（個人情報扱い）
- 入力された電話番号・予約番号は照合用のみ、ログには出さない
- 紐付け失敗時は何が間違っていたか具体的に出さない（不正な照会を防ぐため一律「見つかりません」）
- id_token は `LineIdTokenVerifier` で `aud=login_channel_id` まで検証

## 11. 既存 link token フローとの併用

既存の予約完了→メール→`/line/liff/link/{token}` フローはそのまま残置。
両フローが衝突しないよう、`customer_line_contacts.line_user_id` の UNIQUE 制約により
**先に紐付いた方が優先**される。後発はエラー応答で離脱。
